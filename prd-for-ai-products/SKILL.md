# PRD-for-AI-Products（write-ai-prd）：专为AI应用产品编写的PRD生成指令包

将用户模糊的AI产品构思，通过系统化的分批提问逐一锁定AI专属决策维度，最终输出一份专业PRD——一份结构清晰、决策完整、可直接落地的AI产品方案。

> 命名风格参考：`功能描述 (english-code)`，本指令英文代号 **write-ai-prd**，便于在 Agent 技能市场检索与引用。

## 适用范围

本技能适用于AI产品（AI原生/AI增强/AI基础设施/AI+领域融合）。如果产品不涉及AI核心能力，应使用通用版PRD技能。

## 安全边界

本指令只做「提问 + 生成 PRD 文档」，遇到以下边界会停下来问你，绝不擅自动作：

- **不替你做业务决策**：只引导、给推荐选项并说明理由，最终拍板权在你（如 AI 产品形态、商业化模型、是否向上取级）。
- **不编造事实**：信息缺失标 `[TODO]`，不确定标 `[未知]`，交付前汇总，绝不虚构数据 / 市场 / 竞品数字。
- **不执行危险操作**：不删除文件、不提交 git、不发外部请求、不调用任何写操作——PRD 里的「写操作需人工确认」是给产品的设计建议，不是本指令会执行的动作。
- **不越权生成**：只在你给的 AI 产品构思范围内生成 PRD；若发现需求实为传统非 AI 产品，会建议改用通用 PRD 流程而非硬套。
- **长任务可中断**：任何一步你都可以喊停、改方向、要求重来，不会偷偷往前跑。

## 核心差异：为什么AI产品需要专属PRD

| 维度 | 通用PRD | AI产品PRD（本技能） |
|------|--------|-------------------|
| 提问维度 | 通用维度 | 15个核心维度中**6个替换为AI专属维度** + **AI深度增强维度（条件触发）** |
| PRD结构 | 通用结构 | 15章结构，**含4个AI专属章节** + **评估验收/Prompt规格/异常降级等AI深度章** |
| 竞品对标 | 通用维度 | 增加**AI能力维度对标** |
| 商业分析 | 通用商业模型 | 增加**算力经济学分析** |
| 质量可判定 | -- | 交付前**PRD质量星级自评（★★★★★）**，对标成熟技能可判定质量 |

## 核心原则

1. **先问后写，绝不抢先动笔**——PRD的质量不在文笔，在决策的完整性和一致性。任何一个未确认的决策点，都可能导致整份PRD的返工
2. **AI能力边界是最重要的决策**——它决定了产品架构、UX、成本、风险的方方面面
3. **每一段PRD内容都应体现决策质量**——不堆砌AI术语，而是展示对AI产品独特性的深度理解

## 整体流程

1. **接收构思** → 理解用户的AI产品模糊想法
2. **范围定级 + 分批提问** → 先定级 L1–L4 并让用户决策是否向上取（L4 直接对标），再分5批（每批3个维度，每个维度展开若干子问题）锁定15个核心决策维度
3. **AI深度增强维度**（条件触发）→ 根据产品类型补充评估验收/Agent行为/工具接口/交互模式
4. **编写PRD** → 全部决策确认后，输出含AI专属章节的完整PRD
5. **校验覆盖 + 质量自评** → 逐项核对决策点 + **PRD质量星级自评（★★★★★）** + 缺口扫描
6. **交付** → 展示校验与评分摘要 → 用户确认 → 输出完整PRD + 双维度亮点提炼

## Step 1：接收与理解构思

用户首次描述AI产品想法时，认真倾听并提炼：
- 产品的核心AI能力是什么？（生成/理解/推荐/决策/执行？）
- AI在产品中的角色？（核心引擎 vs 能力加持）
- 用户的模糊程度——是对AI有清晰判断还是完全不确定？

**不要做的事**：此时不展开分析、不给建议、不写任何PRD内容。

---

## Step 2：分批提问（核心步骤）

### 范围定级与模式（动手前提问前先定级）

> 借鉴成熟 PRD 技能的「复杂度感知」机制：先判断这份 PRD 的复杂度与用途，再决定提问深度与文档详略——避免小需求被硬写成大而全，也避免大需求被轻描淡写。

**复杂度分级（L1–L4）**：

| 级别 | 适用场景 | 提问策略 | 文档详略 |
|------|---------|---------|--------|
| L1 微需求 | 单点功能 / 一句话想法 | 仅批次1+2关键项，其余默认推荐 | 精简版，聚焦能力边界 + 评估 |
| L2 小需求 | 明确场景、范围可控 | 批次1–3 主干，跳过非相关维度 | 标准 15 章裁剪版 |
| L3 中需求 | 完整产品、多模块 | 全 5 批提问，AI增强维度全触发 | 完整 15 章 + AI专属章 |
| L4 大需求 / 平台级 | 生态 / 多端 / 多 Agent | 全 5 批 + 增强维度 + 架构深挖 | 完整版 + 附录扩展 |

**模式标签（决定侧重与呈现）**：

- `[应聘]`：作为作品展示，重方法论外显，实操细节可取舍
- `[落地]`：给研发 / 测试执行，重行为定义 / 边界 / 接口 / 评估集，可落地优先
- `[精简]`：快速产出，跳过非必要维度，输出骨架 + 关键点
- `[移动优先]`：移动端场景，补充端侧推理 / 离线 / 交互约束

> 定级与模式在 Step 1 接收构思后由 Agent 与用户对齐确认，写入 PRD 文档头部的「文档性质」字段。

**定级后的决策权交给用户（关键）**：Agent 给出初始复杂度定级后，必须明确询问用户——「是否按更高标准输出（向上取级）？」例如初始识别为 L2，询问「是否按 L3 标准输出？」。用户决策后锁定最终级别与文档详略。
> **特例**：若初始识别即为 **L4**，则无需问询，直接对标 L4 全量输出。

### 提问节奏规范

- 每批3个维度，聚焦同一主题；每个维度展开若干子问题（含选项表），降低用户认知负荷
- 选项设计：有明显优劣时直接推荐；存在创新可能时加"其他/融合方案"选项
- 每批回答后用引用格式锁定核心决策，确保双方对齐
- AI技术术语首次出现时附通俗解释（如"RAG（检索增强生成，即AI先查资料再回答）"）
- 对AI技术不熟悉的用户，提供"让我来推荐"选项，由Agent给出判断并说明理由

### 五批提问维度

> 每个维度的详细问题模板，在提问前读取 `references/ai-decision-framework.md`。

**批次1：产品定位与目标用户**（含AI形态+AI监管）
- 产品核心定位 + 目标（目标必须落到**业务数字**如解决率/满意度，**勿把动作当结果**——"上线一个AI客服"是动作不是结果） + AI产品形态（AI原生/AI增强/AI基础设施/AI+领域融合）
- 目标用户优先级 + AI认知水平
- 优先市场 + AI监管环境差异

**批次2：AI能力边界与交互**（核心批次）
- AI能力边界定义（能力范围/质量下限/可控性/延迟容忍/渐进增强）
- 命名与品牌调性
- 平台形态 + 端侧/云侧推理决策

**批次3：技术栈、商业与数据**
- AI技术栈与模型策略（模型来源/类型/更新/部署）
- 商业化模型 + 算力成本结构
- 数据飞轮与AI自增强（数据收集/反馈闭环/自增强/合规）

**批次4：AI UX、社区与安全**
- AI UX范式与交互设计（范式选择/修改机制/AI透明度/AI人格）
- 社区与社交机制
- AI安全、伦理与合规（内容安全/幻觉/版权/隐私/偏见/监管/IP保护）

**批次5：技术架构与PRD呈现**
- 技术架构 + AI工程架构深度（中间层/评估/部署/可观测性）
- 竞品锚定 + AI能力维度对标
- PRD呈现定位（立项/实操/应聘）

### 提问中的灵活处理

- 已明确的决策直接确认，不必重复提问
- 不适用的维度跳过并说明原因
- 用户回答揭示新子维度时可追加1-2个追问
- **发现用户对AI技术不熟悉**：在选项中增加"让我来推荐"选项，由Agent基于产品特征给出判断并说明理由，用户只需确认

### AI深度增强维度（条件触发）

> 上述15个维度是**所有AI产品的核心骨架**。以下维度按产品类型**条件触发**，补全笔记方法论中Agent PRD的"八大块"缺口——尤其是笔记强调"AI产品特有、常规PRD写不出的"三块：**评估与验收、Prompt/指令规格、异常与降级**。

| 触发条件 | 增强维度 | 对应笔记八大块 | 必要性 |
|---------|---------|--------------|-------|
| **所有AI产品** | 评估与验收策略 | 08 评估与验收 | 必答——概率输出必须靠评估衡量好坏 |
| **所有AI产品** | 六个关键交互模式 | AI交互设计六模式 | 必答——输出不确定/多轮/有等待感/会出错 |
| **Agent类产品**（自主执行/调工具） | Agent行为定义（场景→期望行为） | 03 Agent行为定义 | 必答——把概率行为写成可预期表现 |
| **Agent类产品** | 工具与接口（RAG/FC/MCP） | 06 工具与接口 | 必答——列清每个工具/参数/权限 |
| **生成类/知识类** | Prompt/指令规格 + 知识源与更新 | 04 Prompt规格 / 05 知识与数据 | 必答——Prompt即功能实现，知识过时头号错误源 |

触发判断在批次5（技术架构与PRD呈现）收口时由Agent明确告知用户："本产品属于[X]类型，因此补充触发以下增强维度"，避免与15核心维度混淆。

详细提问模板见 `references/ai-decision-framework.md` 的"AI深度增强维度"章节。

## Step 3：编写PRD

**前提**：5批提问全部完成，所有决策已锁定；Step 2 已完成范围定级、用户决策与模式确认。

1. 读取 `references/ai-prd-template.md` 获取AI产品PRD标准结构
2. 根据全部决策填充内容，并套用以下撰写原则与写作机制
3. 撰写原则（13条）：
   - 关键洞察、策略、判断必须加粗
   - 表格优先
   - 每章开头用引用格式提炼核心结论
   - 流程图用文本箭头描述
   - 应聘作品须注明
   - **AI能力边界用「产品选择」而非「技术限制」表述**
   - **算力成本必须量化（至少给出量级）**
   - **AI幻觉必须正面回应，有明确应对策略**
   - **AI术语首次出现时附通俗解释**
   - **非目标必须写清**——「这一版不做什么」和「做什么」同等重要，划界防跑偏
   - **评估验收必须可量化**——给出评估集思路与上线门槛值（如幻觉率 < 2%），不靠「感觉还不错」验收
   - **异常降级必须有方案**——AI 答不了 / 答不准 / 系统故障时如何优雅兜底，是 AI 产品的及格线
   - **Prompt / 知识必须进文档**——核心逻辑不藏代码里，版本管理、责任人、更新频率落到纸面
4. 写作机制（吸收成熟 PRD 技能，提升可执行性与可信度）：
   - **方法论外显（💡 callout）**：每个关键决策点旁用 `> 💡 方法论：` 标注「依据什么 / 为什么这么做」，让 PRD 成为可学习的决策档案
   - **[TODO] / [未知] 标注规则**：信息缺失标 `[TODO: …]`，不确定事实标 `[未知: …]`，**严禁编造**；交付前汇总所有 [TODO] / [未知] 进缺口清单
   - **硬约束 / 推荐默认 / 发挥空间 三分法**（写给实现 Agent）：用 `🔒硬约束` 标不可改项、`✅默认` 标推荐项、`🟡发挥` 标留给实现发挥的项
   - **数值指标 + 降级阈值**：关键 AI 指标（幻觉率 / 解决率 / 延迟）必须给目标值与降级阈值（如「幻觉率 < 2%，超 5% 触发人工复核」）
5. 输出完整PRD（Markdown格式，包含全部15章 + Agent条件触发章16/17）

## Step 4：校验覆盖 + PRD质量星级自评

PRD完成后，**四重校验 + 质量自评**：

**校验1：决策点覆盖**
逐一核对分批提问中确认的所有决策点（含AI深度增强维度）是否在PRD中有对应内容：

| # | 决策点 | PRD覆盖位置 | ✅/❌ |

如有遗漏，立即补充后再交付。

**校验2：PRD质量星级自评（★★★★★，借鉴成熟技能的可判定质量机制）**

交付前对PRD按以下4个维度各打1-5星，并展示给用户——让"质量"从主观感受变为可判定信号：

| 评估维度 | 1★（弱） | 5★（强） | 本PRD评级 |
|---------|---------|---------|----------|
| **决策完整度** | 多处未确认就动笔 | 15核心维度+增强维度全锁定 | ⭐⭐⭐⭐⭐ |
| **AI能力深度** | 只写"用AI"无边界/成本/飞轮 | 边界/算力/飞轮/评估/安全全到位 | ⭐⭐⭐⭐⭐ |
| **可执行性** | 写的规则无法落地 | 行为+边界+兜底+评估集可交付 | ⭐⭐⭐⭐⭐ |
| **风险可控性** | 幻觉/越权/合规无应对 | 异常降级+安全合规闭环 | ⭐⭐⭐⭐⭐ |

> 任一维度低于3★，先修订再交付。星级不是装饰——它强制Agent自检"这份PRD是否真的解决了AI产品的核心问题"。

**校验3：缺口与待完善扫描**（吸收成熟 PRD 技能的自检机制）

逐维度扫描完整性，标记状态并列入交付摘要的「待完善清单」：

| 扫描维度 | 检查项 | 状态 |
|---------|--------|------|
| 决策完整性 | 15 核心维度 + AI增强维度是否全确认 | 🔴 / 🟡 / 🟢 |
| AI专属覆盖 | 4 个 AI 章 + AI深度增强子章是否齐 | 🔴 / 🟡 / 🟢 |
| 边界与兜底 | 能力边界 / 异常降级 / 写操作闸是否明确 | 🔴 / 🟡 / 🟢 |
| 评估可判定 | 评估集 / 指标 / 上线门槛是否量化 | 🔴 / 🟡 / 🟢 |
| 数值量化 | 算力成本 / 延迟 / 质量是否给量级 | 🔴 / 🟡 / 🟢 |
| 缺口标注 | [TODO] / [未知] 是否汇总、无编造 | 🔴 / 🟡 / 🟢 |

- 🔴 必补（缺失会导致 PRD 不可用或存在风险）
- 🟡 建议补（影响质量但不阻塞交付）
- 🟢 已完善

四重校验（决策点 + 质量星级 + 缺口扫描）均通过后再交付。

---

## Step 5：确认与交付

> **评审对齐（建议）**：交付前建议拉开发/测试一起评审，确认 PRD 可落地、可验证——文档经评审对齐才能真正指导开发与测试。

1. 向用户展示**校验与评分摘要**（决策点覆盖数 + 4维质量星级 + 🔴🟡🟢 待完善清单与 [TODO]/[未知] 缺口汇总）
2. **确认门**：询问用户"校验与评分通过，是否交付？有需要调整的地方吗？"
3. 用户确认后，输出完整PRD（Markdown格式，用户可自行保存为文件）
4. 交付后提供**双维度亮点提炼**：
   - 产品亮点（3条）：PRD中产品构思的核心竞争力
   - AI 特性亮点（3条）：PRD中体现的AI产品差异化能力

## 示例

### 示例1：AI原生产品

> **用户输入**："我想用某游戏IP资产做一款AI即时生成小游戏的应用，竞品没有这些IP。"

**Agent判断**：没有AI产品不成立 → AI原生产品，全维度高权重提问。

**提问重心示例**：
- 批次2：AI能力边界——首发做到什么程度？质量下限如何兜底？
- 批次3：模型策略——第三方LLM还是自研？算力成本对应定价策略？
